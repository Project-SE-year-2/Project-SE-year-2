## א. מבוא

### א.1 מטרת המסמך

מסמך זה מתאר את עיצוב התוכנה עבור גרסה 2.0 של מערכת שיבוץ שעות הבחינות. הוא מהווה תיעוד טכני מלא הכולל את ארכיטקטורת המערכת, עיצוב ממשק המשתמש, רכיבי התוכנה, וזרימת הנתונים.

### א.2 קהל יעד

- צוות הפיתוח המטמיע את גרסה 2.0
- מובילי גרסה הסוקרים את הקוד
- סגל הקורס המעריך את הפרויקט

### א.3 היקף המסמך

מסמך זה מכסה את כלל הרכיבים המיושמים בגרסה 2.0, כולל:

- **ממשק גרפי (GUI)** — מסך קלט ומסך פלט מלאים
- **שכבת Presenter** — שירות אפליקציה, DataStore, ניהול שרשורים
- **שכבת אלגוריתם** — מנוע שיבוץ עם Backtracking
- **שכבת פרסרים** — טעינת קבצי קורסים ותאריכים
- **שכבת מודלים** — אובייקטי הדומיין

### א.4 מונחים

| מונח | הסבר |
|------|-------|
| **תקופת בחינות** | טווח תאריכים בסמסטר מסוים בהם ניתן לשבץ בחינות |
| **שיבוץ של בחינה** | השמה של בחינה בתאריך אפשרי בתקופת הבחינות |
| **מערכת בחינות** | שיבוץ אפשרי של הבחינות בכלל הקורסים של התוכניות שנבחרו |
| **MVP** | Model-View-Presenter — דפוס ארכיטקטוני המפריד לוגיקה עסקית מממשק המשתמש |
| **DataStore** | רכיב השומר את מצב האפליקציה בזיכרון ועל הדיסק |

## ב. סקירת ארכיטקטורה

### ב.1 דפוס ארכיטקטוני — MVP

המערכת בנויה לפי דפוס **Model-View-Presenter (MVP)** עם הפרדה קפדנית בין שכבות:

```
┌─────────────────────────────────────┐
│           View Layer (PyQt5)        │
│  MainWindow, InputScreen,           │
│  OutputScreen, Widgets              │
└──────────────┬──────────────────────┘
               │ IAppService (interface)
               ▼
┌─────────────────────────────────────┐
│         Presenter Layer             │
│  AppService (Singleton)             │
│  DataStore, GenerateWorker          │
└──────┬────────────────┬─────────────┘
       │                │
       ▼                ▼
┌─────────────┐  ┌──────────────────────┐
│   Parsers   │  │  Algorithm Layer     │
│ CourseParser│  │  SchedulingEngine    │
│ PeriodParser│  │  BacktrackingSolver  │
│ ProgramParser│ │  ScheduleCombiner    │
└──────┬──────┘  └────────┬─────────────┘
       │                  │
       ▼                  ▼
┌─────────────────────────────────────┐
│         Model Layer                 │
│  Course, ExamPeriod, ExamSchedule   │
│  ProgramRequirement                 │
└─────────────────────────────────────┘
```

### ב.2 עקרונות הארכיטקטורה

| עיקרון | יישום |
|--------|-------|
| **הפרדת אחריות** | ה-View לא מכיר ישירות מודלים — רק עובד מול `IAppService` |
| **Dependency Inversion** | ה-View תלוי ב-interface בלבד (`IAppService`), לא ב-`AppService` ישירות |
| **Singleton** | `AppService` הוא singleton — נקודת גישה יחידה ללוגיקה העסקית |
| **Worker Thread** | החישוב מתבצע ב-`GenerateWorker` בשרשור נפרד כדי לשמור על ממשק רספונסיבי |
| **Generator Pattern** | `generate_stream()` מייצר תוצאות לפי תקופה בחינות — מאפשר הצגה מדורגת |

## ג. עיצוב ממשק משתמש

### ג.1 עקרונות עיצוב UI/UX

#### ג.1.1 רציונל עיצובי

ממשק המשתמש של גרסה 2.0 מבוסס על שלושה עקרונות מרכזיים:

1. **פשטות בקלט** — המשתמש (מזכירות) אמור להגיע לתוצאה ב-3 שלבים: טעינת קבצים → בחירת תוכניות → הפקת שיבוצים.
2. **רציפות ויזואלית** — ניווט חלק בין מסך הקלט למסך הפלט ללא איבוד הנתונים שהוזנו.
3. **רספונסיביות** — פעולות חישוביות ארוכות מתבצעות ברקע; הממשק נשאר פעיל בכל עת.

#### ג.1.2 עקרונות ויזואליים

- **פידבק ברור** — כל פעולה (טעינת קובץ, יצירת שיבוץ) מלווה בהצגת תוצאה ברורה.
- **ניווט חד-כיווני** — Input Screen → Output Screen, עם אפשרות חזרה.
- **שמירת מצב** — נתונים שנטענו נשמרים כל עוד לא הוחלפו, גם לאחר ניווט בין מסכים.
- **הנחיות ברורות** — כל שדה קלט מלווה בתווית ברורה ובאימות.

### ג.5 דוגמאות שימוש (Use Cases)

#### ג.5.1 תרחיש: טעינת נתונים ראשונה

1. המשתמש מפעיל את התוכנה — מוצג מסך הקלט עם שדות ריקים.
2. המשתמש לוחץ "עיין" ליד "קובץ קורסים" ובוחר `courses.txt`.
3. המשתמש לוחץ "עיין" ליד "קובץ תאריכים" ובוחר `dates.txt`.
4. המשתמש לוחץ "טען נתונים" — מוצגת ספינת טעינה לרגע.
5. לאחר הטעינה: רשימת תוכניות הלימוד מתמלאת, ולוח תקופות הבחינות מתעדכן.

#### ג.5.2 תרחיש: עדכון קובץ קורסים קיים

1. הנתונים כבר נטענו ממפגש קודם.
2. המשתמש בוחר קובץ קורסים חדש עם קורסים נוספים ולוחץ "עדכן".
3. המערכת ממזגת את הקורסים החדשים עם הקיימים (ללא מחיקת הנוכחיים).

#### ג.5.3 תרחיש: הפקת שיבוץ

1. המשתמש בחר 2 תוכניות לימוד וסקר את תקופות הבחינות.
2. המשתמש לוחץ "הפק שיבוצים".
3. בזמן החישוב — מוצגת חלונית התקדמות. הממשק מגיב ללחיצות.
4. בסיום — המסך עובר אוטומטית לתצוגת הפלט.

#### ג.5.4 תרחיש: עריכת יום חסום

1. המשתמש לוחץ "ערוך" ליד תקופת בחינות.
2. נפתח חלון עריכה עם לוח השנה של התקופה.
3. המשתמש לוחץ על תאריך 14/02 ומסיר אותו מחוק הבחינות.
4. המשתמש לוחץ "שמור" — השינוי מתעדכן ב-DataStore.

#### ג.5.5 תרחיש: ניווט בין מערכות ושמירה

1. במסך הפלט מוצגת מערכת 1 מתוך 47.
2. המשתמש לוחץ "הבא" מספר פעמים עד למערכת 5.
3. המשתמש לוחץ "שמור שיבוץ זה לקובץ" — נפתח חלון שמירה.
4. הקובץ נשמר; המסך נשאר על מערכת 5.

## ד. עיצוב רכיבים מפורט

### ד.1 שכבת ה-View (PyQt5)

#### ד.1.1 MainWindow

**מחלקה:** `MainWindow(QMainWindow)`

**אחריות:** מחזיק `QStackedWidget` המכיל את כל המסכים. מנהל מעבר בין מסכים ומוודא שמצב ה-DataStore נשמר בין מעברים.

| מתודה | תיאור |
|-------|-------|
| `show_input_screen()` | עוברת למסך הקלט |
| `show_output_screen()` | עוברת למסך הפלט |
| `__init__()` | אתחול, יצירת מסכים, חיבור סיגנלים |

#### ד.1.2 InputScreen

**מחלקה:** `InputScreen(QWidget)`

**אחריות:** מרכז את כל הקלט מהמשתמש לפני הפקת השיבוצים.

| רכיב | Widget | אחריות |
|------|--------|--------|
| טעינת קבצים | `FileLoaderWidget` | קלט נתיב + אימות + טעינה ל-AppService |
| בחירת תוכניות | `ProgramListWidget` | הצגה ובחירה של תוכניות לימוד |
| עריכת תקופות | `PeriodListWidget` | הצגה ועריכה של תקופות בחינות |
| כפתור הפקה | `QPushButton` | הפעלת `GenerateWorker` |

**אינטראקציות מרכזיות:**
- לחיצה על "טען נתונים" → קריאה ל-`AppService.load_data()`
- בחירת/ביטול תוכנית → קריאה ל-`AppService.select_program()` / `deselect_program()`
- לחיצה על "הפק שיבוצים" → יצירת `GenerateWorker` והפעלתו
- סיגנל `generation_complete` מ-Worker → ניווט ל-`OutputScreen`

#### ד.1.3 OutputScreen

**מחלקה:** `OutputScreen(QWidget)`

### ד.4 שכבת האלגוריתם

#### ד.4.1 SchedulingEngine

**מחלקה:** `SchedulingEngine`

**אחריות:** תיאום תהליך השיבוץ עבור כלל התקופות. מפעיל `BacktrackingSolver` לכל תקופה ומעביר תוצאות ל-`ScheduleCombiner`.

**זרימה:**
```
SchedulingEngine.generate()
  └── for each ExamPeriod:
        ├── CourseOrderingHeuristic.order(courses)
        ├── ForwardChecker.initialize(period)
        ├── BacktrackingSolver.solve(ordered_courses, period)
        │     ├── ConstraintIndex.get_obligatory_groups()
        │     └── ConstraintValidator.validate(assignment)
        │           └── BasicVersionValidator.collides(exam1, exam2)
        └── yield (period_id, List[ExamSchedule])
  └── ScheduleCombiner.combine(all_period_results)
```

#### ד.4.2 BacktrackingSolver

**אחריות:** אינומרציה של כל שיבוצי הבחינות האפשריים בתקופה נתונה באמצעות חיפוש עם Backtracking.

- משתמש ב-`ForwardChecker` לגיזום ענפים בלתי אפשריים מוקדם
- משתמש ב-`ConstraintValidator` לאימות כל השמה

#### ד.4.3 ScheduleCombiner

**אחריות:** שילוב תוצאות מכל התקופות למערכות בחינות שלמות.

## ז. דרישות ביצוע וביטחון

### ז.1 ביצוע

| דרישה | יישום |
|-------|-------|
| **ממשק רספונסיבי** | `GenerateWorker` מריץ את החישוב בשרשור נפרד (QThread) |

| **זרמת תוצאות** | `generate_stream()` מייצר תוצאות לפי תקופה — חוסך זיכרון ומאפשר הצגה מדורגת |

### ז.2 אמינות

- **אימות קלט** בשלב ה-FilePathValidator לפני כל טעינה
- **טיפול בשגיאות** בפרסרים — שגיאת פורמט מציגה הודעה ברורה ואינה קורסת את התוכנה
- **GenerateWorker** מאזין לשגיאות ומפרסם `error_occurred` signal למסך הקלט

### ז.3 תחזוקה ועתיד (גרסה 3.0)

הארכיטקטורה מוכנה להרחבת גרסה 3.0:
- `ICollisionValidator` — ניתן להחליף אסטרטגיית בדיקת התנגשויות
- `IOutputWriter` — ניתן להוסיף פורמטים נוספים לייצוא
- `IAppService` — ניתן להחליף את המימוש ב-Mock בבדיקות
- מיון וסינון מערכות בחינות (דרישה 2.5) — מוכן להוספה ב-`AppService`
